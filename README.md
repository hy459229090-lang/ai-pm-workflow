## 这套方法解决什么问题

产品经理的日常被四个问题反复消耗：

> **信息碎片化** — PRD 在文档工具里，数据在 BI 平台上，设计在 Figma 里，决策在会议纪要里。每次推进一个需求，先花半小时找齐上下文。

> **重复劳动** — 写 PRD 的结构大同小异，取数的 SQL 模式相似，埋点方案的格式固定。但每次都从零开始，因为"上次那个文档找不到了"。

> **上下文丢失** — 上周讨论的结论、上个版本的设计取舍、某个指标异常的原因……这些信息存在于某个人的脑子里，没有沉淀。换一个人接手，一切从头来过。

> **协作低效** — 研发问"这个边界怎么处理"，产品去翻文档；算法问"这个特征怎么定义"，产品去找数据。PM 变成了信息中转站，而不是决策者。

**核心思路**：把你的工作流、涉及到的具体信息、上下文、每一项原子化能力，以及必要的协作信息，通过 AI 串联起来。让 AI 成为你的工作记忆和执行助手，而你专注于判断和决策。

## 三条核心理念

<table>
<tr>
<td width="33%" valign="top">

### 1. 上下文工程 > 模型选择

大多数人在纠结用哪个模型、调什么参数。但实际效果 80% 取决于你**喂给 AI 什么信息**。

同一个模型，不告诉它你的项目背景，它给你通用的建议；告诉它你的团队构成、技术栈、用户画像、历史决策，它给你**可执行的方案**。

</td>
<td width="33%" valign="top">

### 2. 渐进式搭建

你不需要一步到位搭 14 个 Agent。

- **Day 1**：1 个文件 → 立刻有用
- **Week 1**：3 个 Skill → 日省 30 min
- **Month 1**：上下文 + Review → 质量可控
- **之后**：按需扩展 → 复杂任务跑通

每一步独立可用，随时可以停下来。

</td>
<td width="33%" valign="top">

### 3. AI 是副驾驶

AI 做：收集信息、组织文档、模式化执行、多角度审视。

人做：审批计划、做决策、解除阻塞、判断质量。

**三个不可跳过的关卡**：
1. 审批执行计划
2. 做决策 / 解除阻塞
3. Review 最终产出

</td>
</tr>
</table>

## 贯穿案例：毛球 App

为了让每篇教程都有具体场景，本系列使用一个虚构的产品案例：

> **毛球** — 宠物主人的内容社区 + 本地服务平台。分享宠物日常、发现附近服务、AI 健康问答、用品团购。

```
                     ┌─────────────┐
                     │   毛球 App   │
                     └──────┬──────┘
            ┌───────────────┼───────────────┐
            v               v               v
      ┌───────────┐   ┌───────────┐   ┌───────────┐
      │  内容社区   │   │  附近服务   │   │  AI 问答   │
      │ 图文/视频  │   │ 医院/寄养  │   │ 健康知识库 │
      └───────────┘   └───────────┘   └───────────┘
            │               │               │
            v               v               v
      推荐算法 PRD      LBS 服务匹配      RAG + Prompt
```

## 快速开始

**你需要**：[Claude Code](https://claude.ai/code)（CLI / Desktop / IDE 插件均可） + 一个你正在做的项目 + 15 分钟

```bash
# 1. 克隆本仓库
git clone https://github.com/hy459229090-lang/ai-pm-workflow.git

# 2. 阅读第 0 篇，理解核心理念
#    docs/00-philosophy.md

# 3. 跟着第 01 篇，在你自己的项目里创建 CLAUDE.md
#    docs/01-claude-md.md → templates/CLAUDE.md.template
```

不需要安装任何依赖，不需要配置环境。**本仓库就是一套文档和模板。**
