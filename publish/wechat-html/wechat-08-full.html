第 08 篇：模板与规范 — 让产出一致（微信公众号版）

一句话：模板不是限制创造力的枷锁，是让 AI 产出从一开始就像你写的东西。

一个差点让我崩溃的瞬间

第 05 篇讲了 Agent Team——让多个 AI 分工协作完成复杂任务。

毛球 App（一个宠物社区 + 本地生活服务平台）做推荐 feed 改版时，我拉了产品 Agent、技术 Agent、数据 Agent 一起干活。

效果确实好：PRD、技术方案、埋点方案、测试用例，几小时内全部出初稿。

但用了 Agent Team 一段时间，我发现一个新问题：

产出越来越多，风格越来越乱。

产品 Agent 写的推荐 PRD 是一个风格，技术 Agent 写的召回方案是另一个风格，数据 Agent 出的埋点文档又是第三个风格。

毛球推荐项目出来的三份文档，读起来像是三个不同公司写的。

前端研发拿到 PRD 说："这个推荐卡片排序逻辑跟以前的 PRD 格式不一样，我没看懂'相关性得分'怎么计算。"

测试拿到埋点方案说："这个事件命名跟以前的不一样，recommend_expose 和 rec_exposure 是一个东西吗？"

产出质量没问题，但一致性太差。 每次有人读新文档，都要重新适应格式、重新理解术语、重新找重点在哪。 更关键的是：这不是 AI 的问题，是我的问题。

我没有给它们统一的模板和规范。我让每个 Agent 按自己的方式写——那当然写出来不一样。

模板 vs 规范：不是同一个东西

最初我把"模板"和"规范"混为一谈。后来发现它们是两件事。

模板：填空式的结构骨架

模板回答的问题是："这份文档应该长什么样？"

一份好的 PRD 模板包含：

# [需求名称] ## 背景 - 为什么做这个需求 - 解决了什么问题 ## 目标 - 业务目标（可量化） - 用户目标 ## 方案 - 功能描述 - 交互流程 ## 排期 - 里程碑 - 风险点

模板的价值：不需要每次从零开始想结构。 你知道重点在哪、顺序是什么、缺了什么一眼就能看出来。

规范：约束性的规则

规范回答的问题是："写的时候有哪些必须遵守的规则？"

比如：

● 命名规范：所有 PRD 文件名用 [模块]_[功能]_PRD.md

● 字段标准：每个功能必须包含"成功指标"，且必须是可量化的数字

● 术语统一："用户"不等于"账号"，"埋点"不等于"日志"

规范的价值：不同人（包括 AI）写出来的东西，能直接对齐。

两者的关系

模板是骨架，规范是肌肉记忆。
模板规范
回答什么结构长什么样写的时候遵守什么规则
灵活性高（可以选不同模板）低（必须遵守）
例子PRD 模板、埋点方案模板命名规范、术语表
两者缺一不可。 只有模板没有规范——结构统一了，但内容质量参差不齐。只有规范没有模板——规则都懂，但每次都要重新组织。

我的模板体系

分享一下我实际在用的模板。不是最完美的，但都经过真实场景验证。

1. PRD 模板

# [模块]_[功能]_PRD > **AI 导航**：收到此文档时，请先确认已读取项目文档中的相关文档路径。 ## 背景 一句话说明为什么要做。不写"为了提升用户体验"这种废话。 ## 目标 | 类型 | 指标 | 当前值 | 目标值 | |------|------|--------|--------| | 业务 | | | | | 用户 | | | | ## 用户场景 用用户视角描述，不用功能视角。 > "作为一个养猫的人，我想..." ## 方案 ### 核心流程 1. 第一步 → 第二步 → 第三步 ### 异常流程 - 如果 A 失败，走 B ### 非功能需求 - 性能、兼容性、安全性 ## 埋点 | 事件 | 触发时机 | 属性 | 上报端 | |------|---------|------|--------| ## 排期 | 里程碑 | 负责人 | 预计完成 | |--------|--------|----------|

2. 埋点方案模板

# [模块]_埋点方案 ## 事件列表 | 事件名 | 触发时机 | 属性 | 示例值 | 上报端 | |--------|---------|------|--------|--------| ## 属性字典 | 属性名 | 类型 | 说明 | 枚举值 | |--------|------|------|--------|

3. 数据报告模板

# [主题] 数据报告 > 报告周期：YYYY-MM-DD ~ YYYY-MM-DD ## 核心指标 | 指标 | 本期 | 上期 | 变化 | |------|------|------|------| ## 趋势分析 一句话结论 + 图表 + 异常点说明 ## 洞察与建议 - 发现了什么 - 建议做什么

模板是怎么来的

这些模板不是一开始就设计好的。是这么演化的：

1. 第一次写：从零开始，边写边想结构

2. 第二次写：复制第一次的，删掉内容只留结构 → 模板 1.0

3. 第三次到第五次：每次写完发现"这个地方应该有，但上次忘了" → 加到模板里

4. 稳定下来：大概 5-10 次之后，模板就不再变了

不要一开始就追求完美模板。 用几次自然就知道缺什么。

我的规范体系

模板是公开的（放在 templates/ 目录），规范更多是隐性的——写在 CLAUDE.md 里，让 AI 每次都能看到。

命名规范

| 文件类型 | 命名格式 | 示例 | |---------|---------|------| | PRD | `[模块]_[功能]_PRD.md` | `推荐_feed_卡片排序_PRD.md` | | 埋点方案 | `[模块]_埋点方案.md` | `推荐_埋点方案.md` | | 数据报告 | `[主题]_数据报告.md` | `DAU 周报_2026W15.md` | | 调研 | `[类型]_[主题].md` | `竞品_推荐功能.md` |

写作规范

1. 说人话：不用"赋能""抓手""沉淀"这种词 2. 具体优先："毛球社区发布率从 3.2% 涨到 5%"比"用户参与度显著提升"好 10 倍 3. 承认不确定性：不知道就写"待定"，不要编 4. 一页原则：核心内容控制在一页内，详情放链接 5. 日期必写：所有文档必须有创建/更新日期

术语规范

| 术语 | 定义 | 禁止用 | |------|------|--------| | 用户 | 注册账号的真实人 | 不要和"账号"混用 | | 埋点 | 前端行为数据采集 | 不要叫"日志" | | DAU | 日活跃用户数（去重） | 不要和"PV"混用 | | 召回 | 从候选集中筛选用户感兴趣的内容 | 不要叫"推荐" |
术语表的价值：当 AI 说"召回率"的时候，它知道是"从候选集筛选"的意思，而不是"把用户拉回来"的意思。

模板/规范 vs 创造力

一个常见误区：模板和规范会限制创造力。

我的经验恰恰相反。

模板和规范解决的是"低级问题"——格式、命名、术语、结构。这些问题解决了，你才能把精力放在真正需要创造力的地方。

没有模板的时候，写 PRD 的精力分配：

● 30%：想结构（应该先写背景还是先写方案？）

● 20%：调格式（这个表格怎么排好看？）

● 50%：写内容（真正的产品思考）

有了模板之后：

● 5%：选模板

● 5%：填格式

● 90%：写内容

模板不是限制，是解放。

怎么开始

第 1 步：不要一开始就搞全套

最常见的错误：花一周时间设计"完美模板体系"，然后发现没人用。

正确的做法：从你最常写的那份文档开始。

第 2 步：模板 = 你写过的最好的那份文档的结构

不要凭空设计模板。把你写过的最好的那份同类文档，删掉内容只留结构——这就是你的 1.0 模板。

第 3 步：规范 = 你反复说过的那些话

想想你每次 review 产出的时候，反复说的那些话：

● "这个命名不对，应该是 XXX"

● "这个表应该加上 XXX 列"

● "这个词不要用，我们叫它 XXX"

把这些话写成规则，就是规范。

第 4 步：把模板和规范放进 CLAUDE.md

这一步最关键。模板和规范如果只在你脑子里，AI 永远不知道。

效果对比

指标没有模板/规范有模板/规范
新文档启动时间10-15 分钟1-2 分钟
格式调整时间每次都要调几乎为零
术语一致性经常混用自动统一
AI 产出可用率~60%~90%

核心洞察

模板和规范不是管理工具，是沟通工具。

模板让不同的人（包括 AI）用同样的结构思考；规范让不同的人用同样的语言交流。

但模板和规范不是银弹——它解决的是"一致性问题"。

如果内容本身没价值，再好的模板也救不了。模板和规范的价值在于：把需要创造力的部分和不需要的部分分开，让创造力不被琐事消耗。
