# AI 编程助手总是找错文件？你需要一张 WORKSPACE_MAP

> 适合发布为知乎专栏文章，也可回答「AI 工具怎么管理大项目的文档」「Claude Code 怎么让 AI 读对文件」等问题。

---

## 结论

在项目根目录放一个 `WORKSPACE_MAP.md`，核心是一张**任务路由表**——告诉 AI "做某类任务时，直接读哪些文件"。20 分钟写完，AI 的文件定位速度从 30 秒降到 2 秒，准确率从"经常找错"变成"精准命中"。

配合上一篇的 CLAUDE.md，AI 就同时具备了"理解项目"和"定位文档"两项能力。

---

## 问题在哪

上一篇我们在项目根目录放了 CLAUDE.md，让 AI 理解项目背景。但实际使用中会发现一个新瓶颈：**AI 知道项目是什么，但不知道东西在哪。**

当你说"对比埋点方案和 PRD 的定义"，AI 的行为是：

1. 扫描项目目录树
2. 靠文件名猜测哪些文件相关
3. 逐个读取，直到找到目标
4. 可能读到旧版本或错误文件

项目文件少于 10 个时还能凑合。一旦超过 30-50 个文件（大多数正式项目），AI 的定位准确率急剧下降。

根本原因：AI 缺少**文档结构的元信息**——每个文件是什么、放在哪、跟什么任务相关。

## WORKSPACE_MAP 的设计

### 核心：任务路由表

WORKSPACE_MAP 最关键的部分不是文件列表，而是**任务路由表**：

```markdown
| 当前任务 | 直接读取这些文件 |
|----------|-----------------|
| 写推荐系统 PRD | docs/prd/推荐系统_PRD.md → docs/prd/推荐卡片规范.md |
| 做埋点方案 | docs/data/推荐_埋点方案.md |
| 拉数据/写 SQL | scripts/data/ → docs/data/数据字典.md |
```

AI 收到任务后，先查路由表匹配任务类型，直接跳到目标文件。跳过了"遍历目录 → 猜测文件名 → 逐个尝试"的过程。

`→` 符号表示阅读顺序——先读主文档，再读补充文档。这解决了"AI 不知道哪个是最新的/权威的"问题。

### 三层结构

| 层次 | 内容 | AI 怎么用 |
|------|------|----------|
| 任务路由表 | "做 X → 读这些文件" | 最先看，直接匹配任务跳转 |
| 目录导航 | 每个目录的用途、文件数、活跃度 | 任务不在路由表里时，缩小搜索范围 |
| 文件索引 | 核心文件的描述 + 关键词 | 通过关键词匹配精确定位 |

### 效果对比

| 场景 | 无 MAP | 有 MAP |
|------|--------|--------|
| "看推荐相关文档" | 扫描全目录，30秒，可能找错 | 查路由，2秒，精准 |
| "这份 PRD 是最新的吗" | AI 无法判断 | MAP 有更新日期 |
| 50+ 文件的项目 | AI 频繁迷路 | 依然精准 |

## 关键词机制

文件索引里的关键词字段是一个重要细节：

```markdown
| 文件 | 描述 | 关键词 |
|------|------|--------|
| 推荐系统 PRD | 整体方案 | 推荐/算法/冷启动/AB |
```

当你说"帮我看冷启动相关的"，AI 通过关键词 `冷启动` 直接定位到推荐系统 PRD。

如果只靠文件名 `推荐系统_PRD.md`，AI 需要先读文件内容才能判断它是否包含冷启动相关内容。关键词把这个判断前置了。

---

## 维护策略

MAP 的价值取决于它是否保持最新。过时的地图比没有地图更糟——AI 会被导向错误文件。

最小成本的维护方式：在 CLAUDE.md 的写入规范里加一条规则：

```markdown
每次新建文件后，必须同步更新 WORKSPACE_MAP.md 对应条目。
```

如果你用 Claude Code 或 Cursor，AI 在创建新文件时会自动遵守这条规则，帮你维护 MAP。
