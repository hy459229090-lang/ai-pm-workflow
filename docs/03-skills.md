# 第 03 篇：把重复工作变成一句话 — Skill 与原子化能力

> 一句话：Skill 不是 prompt 模板，是给 AI 增加的手段。手段越多，AI 能帮你完成的工作流就越完整。

---

## 我是怎么走到这一步的

搭完 CLAUDE.md 和 WORKSPACE_MAP 之后，AI 确实好用多了——它理解项目、能找到文件、产出质量也上了一个台阶。

但用了两周，我发现一个新问题：**我开始反复做同样的事情。**

"帮我查一下昨天的 DAU。"——每次都要说一遍数据源在哪、表结构是什么、过滤条件怎么写。

"帮我生成一张概念图。"——每次都要说一遍调哪个接口、用什么模型、保存到哪个目录。

"帮我按模板写一份 PRD。"——每次都要说一遍模板在哪、哪些字段必填、格式规范是什么。

AI 有记忆了，但**它没有技能。** 它知道你的项目是什么（CLAUDE.md），知道文件在哪（WORKSPACE_MAP），但每次执行具体操作时，还是要你手把手教。

---

## Skill 是什么：不是模板，是原子化能力

最初我把 Skill 理解成"Prompt 模板"——写好一段提示词，存成文件，需要的时候调用。

这个理解是错的，或者说太浅了。

**Skill 真正的意义是给 AI 增加一种能力。** 就像你给新员工做培训，不是给他一份操作手册让他照着念，而是让他**学会做一件事**——知道调用什么工具、遵循什么标准、产出什么格式。

举个例子：

我有一个 `/draw` 的 Skill。它不是一个画图的 prompt 模板——它定义了：
- 调用哪个图片生成 API
- 有哪些模型可用，各自的限制是什么
- 输出文件应该保存到哪个目录
- 在真正调接口之前，必须先优化 prompt 并让我确认

有了这个 Skill，AI 就"会画图"了。我说"帮我生成一张推荐系统的架构示意图"，它知道调什么接口、什么模型适合、保存到哪里。**我不需要每次都教一遍。**

这就是"原子化能力"的意思：一个独立的、可复用的、AI 能自主执行的技能单元。

---

## 哪些事情适合做成 Skill

不是所有重复操作都该封装。我的判断标准：

**适合做 Skill 的：**
- 流程固定，但每次输入不同（取数、生成图片、按模板写文档）
- 涉及外部工具或 API（不封装的话 AI 不知道怎么调用）
- 有明确的质量标准（格式、字段、校验规则）

**不适合做 Skill 的：**
- 每次都不一样的创造性工作（比如想产品方向）
- 太简单的操作（封装的成本比每次说一句还高）
- 需要大量判断的任务（这种应该拉 Agent Team，后面会讲）

简单说：**Skill 解决的是"怎么做"已经确定的事情。如果"做什么"和"怎么做"都不确定，Skill 帮不了你，你需要的是更高级的协作方式。**

---

## 我的三个核心 Skill

分享我实际在用的三个，都是从"重复做了太多次受不了了"演化出来的。

### 1. 数据取数（`/data`）

**解决什么**：产品经理天天要看数据，每次都要描述表结构、字段含义、过滤条件。

**Skill 定义了什么**：
- 我们的数据平台是什么、怎么连接
- 核心业务表的结构和字段含义（数据字典）
- 常见取数场景的 SQL 模式
- 输出格式规范（表头、排序、时间范围）

**用之前**：
> "帮我查一下过去 7 天的 DAU，用 t_user 表，按 last_login_time 过滤，记得排除测试账号（user_id < 1000），输出按日期排序。"

**用之后**：
> "/data 过去 7 天 DAU"

六个字搞定，因为 AI 已经知道用什么表、怎么过滤、怎么排序。

### 2. 图片生成（`/draw`）

**解决什么**：PRD 里经常需要概念图、交互示意、架构图。以前要开 Figma 画，现在用 AI 生成。

**Skill 定义了什么**：
- 图片生成 API 的地址和认证方式
- 可用模型列表和各自的特点
- 图生图的限制（只有特定模型支持）
- 保存路径和命名规范
- **必须先优化 prompt 再生成**——这条是踩坑后加的，直接用用户的描述去生图效果很差

**怎么用**：
> "/draw 一张推荐系统的数据流向图，从用户行为到推荐结果"

AI 会先把我的描述优化成适合图片生成的 prompt，给我看一遍，确认后才调接口。

### 3. PRD 写作（配合模板）

**解决什么**：PRD 的结构大同小异——背景、目标、方案、排期、风险。每次从零写太慢，但直接填模板又太死板。

**Skill 定义了什么**：
- PRD 模板的位置和结构
- 每个必填字段的含义和写作要求
- 不同类型需求（新功能/优化/修复）的模板差异
- 产出后的自检清单

**关键发现**：PRD Skill 不是"帮我填模板"，而是"按这个标准写，但内容根据输入自主组织"。早期我写得太死——定义了每一段写什么——AI 产出的东西千篇一律。后来我把 Skill 改成只定义**标准和约束**，让 AI 自己组织内容，效果好了很多。

## Skill 和 Agent Team 的边界

这可能是这篇最重要的部分。

**Skill 是一个人的技能，Agent Team 是一群人的协作。**

我做一个简单任务——查个数、画张图、写个小需求——一个 Skill 就够了。AI 调用这个能力，直接出结果。

但当任务变复杂——涉及多份文档产出、需要不同视角评审、有前后依赖关系——单个 Skill 搞不定了。这时候我会判断是否需要拉 Agent Team。

我的判断标准：
- **中间产出超过 2 份**（比如一个需求要出 PRD + 技术方案 + 埋点方案）→ 拉 Team
- **需要不同视角审视**（产品、技术、用户、商业）→ 拉 Team
- **我没法一次 review 完所有产出**（时间/精力不够）→ 拉 Team

反过来，如果是一个独立的、一次性出结果的任务，用 Skill 就够了，拉 Team 反而是浪费。

**这个边界其实我也还在摸索。** 有时候我觉得"这个简单，用 Skill 就行"，做到一半发现需要拉 Team；也有觉得需要 Team 的，后来发现一个好的 Skill 就搞定了。

如果你也在用 AI 工作流，你是怎么判断的？这个我真的很想听不同人的做法。

## 怎么写一个 Skill

以毛球 App 为例，把"取数"封装成 Skill。

### 第 1 步：找到重复模式

回忆你过去一周跟 AI 说过的话，哪些是重复的？把它们列出来。

### 第 2 步：提取共性

这些重复操作有什么共同点？需要的输入是什么、固定的步骤是什么、输出格式是什么？

### 第 3 步：写 Skill 文件

在 `.claude/commands/` 或 `.cursor/skills/` 下创建文件（取决于你用什么工具）。

**不要写成 prompt 模板。** 写成"能力定义"：

```markdown
# /data — 数据取数

你现在具备数据查询能力。

## 你能做什么
- 根据自然语言描述，生成 SQL 查询
- 执行查询并格式化输出结果

## 数据源
- 业务数据库：PostgreSQL，表结构见 docs/data/数据字典.md
- 分析库：ClickHouse，用于大数据量聚合查询

## 规则
- 默认时间范围：近 7 天
- 默认排除测试数据：user_id >= 1000
- 输出格式：Markdown 表格，按日期倒序

## 常见查询模式
- DAU：SELECT date, COUNT(DISTINCT user_id) FROM t_event_log ...
- 留存：...
```

关键区别：这不是告诉 AI"当用户说 X 时回答 Y"，而是告诉 AI **"你现在会做这件事了"**。

### 第 4 步：测试和迭代

用几次真实任务测试。你会发现哪些规则多余、哪些缺失。我的经验是一般迭代 2-3 轮才稳定。

---

## 自查清单

- [ ] 从真实的重复操作出发，不是凭空设计
- [ ] 定义的是"能力"而不是"模板"——AI 有自主判断空间
- [ ] 包含了工具/API 的调用方式（如果涉及）
- [ ] 包含了质量标准和约束，但不过度规定格式
- [ ] 经过 2-3 轮真实使用迭代
