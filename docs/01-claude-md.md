# 第 01 篇：让 AI 理解你的项目 — CLAUDE.md

> 一句话：用一个文件把你的项目背景喂给 AI，让它从"通用助手"变成"项目专家"。

---

## 场景

你刚接手毛球 App 的推荐系统改版，需要写一份 PRD。你打开 Claude Code，说：

> "帮我写一个内容推荐算法的 PRD。"

AI 回了一份 3000 字的文档。你扫了一眼：

- "根据用户兴趣进行个性化推荐"——废话。
- "推荐算法可采用协同过滤或深度学习"——你团队只有 1 个算法工程师，深度学习根本做不了。
- "建议进行 A/B 测试验证效果"——没提你的 DAU 只有 12 万，分桶怎么分都不够显著。

这不是 AI 不行，是它**不知道你的情况**。

现在，你在项目根目录放了一个 CLAUDE.md。同样的问题，AI 的回答变了：

- "基于内容标签 + 简单协同过滤的两阶段推荐，第一版先做召回，排序后续迭代。"——因为它知道你只有 1 个算法。
- "冷启动阶段用宠物档案（品种、年龄）做粗排。"——因为它知道你有宠物档案数据。
- "A/B 测试建议按城市分桶而非用户随机分桶，避免样本量不足。"——因为它知道你的 DAU 量级。

**区别只在于一个文件。**

---

## 原理

### CLAUDE.md 是什么

CLAUDE.md 是放在项目根目录的一个 Markdown 文件。Claude Code 每次启动时会自动读取它，作为所有对话的背景知识。

你可以把它理解为：**给 AI 的项目入职手册**。一个新同事入职，你会告诉他什么？

- 这个项目是做什么的？
- 团队有哪些人，各自负责什么？
- 技术栈是什么？
- 当前在做什么，接下来要做什么？
- 文档放在哪里，怎么找？
- 有什么规矩和惯例？

CLAUDE.md 就是把这些信息写成文件，让 AI 每次都能看到。

### 为什么有效

AI 的能力 = 模型能力 × 上下文质量。

模型能力你控制不了（那是 Anthropic/OpenAI 的事），但上下文质量 100% 由你决定。CLAUDE.md 解决的就是上下文的**基础层**——那些变化频率低、但每次对话都需要的项目背景信息。

没有它，你每次对话都要重复说一遍背景；有了它，AI 开口就带着完整上下文。

### CLAUDE.md 应该写什么

从实际使用经验来看，一个好的 CLAUDE.md 包含这几个板块：

| 板块 | 内容 | 为什么需要 |
|------|------|-----------|
| **项目定位** | 一句话说清楚这是什么产品，给谁用的 | AI 回答的基调和语境 |
| **团队构成** | 多少人、什么角色、各自负责什么 | AI 评估方案可行性的依据 |
| **技术栈** | 前端、后端、数据、部署 | AI 给出的技术建议能落地 |
| **文档结构** | 目录是怎么组织的，什么文件在哪里 | AI 能自己找文件，不用你每次指路 |
| **核心项目** | 当前最重要的项目，关键文档索引 | AI 能快速定位到最相关的上下文 |
| **协作约定** | 团队的工作方式、文档更新规则 | AI 的产出符合团队规范 |
| **写入规范** | 命名规则、目录归属、格式要求 | AI 创建文件时自动遵守 |

不需要全部写满。**先写前三个板块**（项目定位、团队构成、技术栈），就已经比没有好 10 倍了。其他的按需补充。

### 写多少合适

一个常见误区是写太长。CLAUDE.md 会在每次对话时加载，太长会浪费上下文窗口。

| 项目阶段 | 建议长度 | 说明 |
|---------|---------|------|
| 刚起步 | 50-100 行 | 项目定位 + 团队 + 技术栈 |
| 稳定期 | 100-200 行 | 加上文档结构 + 核心项目索引 |
| 复杂项目 | 200-300 行 | 加上写入规范 + 协作约定 |

超过 300 行就该考虑拆分了——用 `AI 导航` 指令引导 AI 去读其他文件（比如 WORKSPACE_MAP.md），而不是全部塞进 CLAUDE.md。

## 实操

跟着做，15 分钟内完成。

### 第 1 步：创建文件

在你的项目根目录创建 `CLAUDE.md`。

```bash
touch CLAUDE.md   # 或者在编辑器里新建
```

### 第 2 步：写项目定位（2 分钟）

回答一个问题：如果一个完全不了解你项目的聪明人问"这是做什么的"，你怎么用三句话说清楚？

```markdown
# 项目名称

项目简介，一两句话。说清楚：做什么、给谁用、核心价值。

> **AI 导航**：收到问题时请优先读 `WORKSPACE_MAP.md` 定位相关文档路径，再精准读取目标文档，避免全量扫描。
```

最后那行 `AI 导航` 是一个效率技巧：告诉 AI 不要盲目搜索整个项目，先看地图。你暂时没有 WORKSPACE_MAP 也没关系，下一篇会讲。先放这行，后续激活。

### 第 3 步：写团队和技术栈（5 分钟）

```markdown
## 团队

| 角色 | 人数 | 职责 |
|------|------|------|
| 产品经理 | 1 | 需求、PRD、数据分析 |
| 前端 | 3 | App + 管理后台 |
| ... | ... | ... |

## 技术栈

- 前端：React Native (App) + React (后台)
- 后端：Python (FastAPI)
- 数据：MySQL + ClickHouse
- 部署：阿里云 K8s
```

写到这里，你的 AI 已经不会再建议你用 Swift 写 iOS 原生了。

### 第 4 步：写文档结构（5 分钟）

把你项目里的关键目录列出来，标注每个目录放什么。

```markdown
## 文档结构

| 目录 | 内容 | 主要读者 |
|------|------|---------|
| `docs/prd/` | 产品需求文档 | 产品、研发 |
| `docs/data/` | 埋点方案和 SQL | 产品、数据 |
| `design/` | 设计稿和标注 | 产品、前端 |
| ... | ... | ... |
```

这一步的价值：当你说"帮我看一下推荐相关的 PRD"，AI 知道去 `docs/prd/` 找，而不是全盘扫描。

### 第 5 步：写核心项目索引（3 分钟）

把当前最重要的 1-2 个项目的关键文档列出来。

```markdown
## 当前核心项目：推荐系统改版

| 文档 | 路径 | 说明 |
|------|------|------|
| 主 PRD | `docs/prd/推荐系统_PRD.md` | 整体方案 |
| 算法方案 | `docs/prd/推荐算法_方案.md` | 召回 + 排序 |
| 埋点方案 | `docs/data/推荐_埋点方案.md` | 行为回收 |
```

**搞定。** 你现在有一个可用的 CLAUDE.md 了。

### 验证效果

打开 Claude Code（或重新开一个会话），问一个和你项目相关的问题。对比有 CLAUDE.md 和没有的回答质量。

## 自查清单

完成后对照检查：

- [ ] 文件放在项目根目录，文件名是 `CLAUDE.md`（大写）
- [ ] 项目定位写了，一个外人能看懂这是做什么的
- [ ] 团队构成写了，AI 能判断方案的可行性
- [ ] 技术栈写了，AI 不会建议你用项目里没有的技术
- [ ] 至少列了核心目录结构，AI 能找到关键文件
- [ ] 总长度在 50-300 行之间，没有塞太多细节
- [ ] 不包含敏感信息（密码、token、内部域名等）

## 常见问题

**Q：我用的是 Cursor / GPT，不是 Claude Code，能用吗？**

A：思路完全一样。Cursor 会读取项目根目录的 `.cursorrules` 文件，作用等价于 CLAUDE.md。GPT 的话可以在 Custom Instructions 里贴项目背景，或者每次对话开头引用一个固定文件。核心不是文件名叫什么，而是**让 AI 每次对话都能自动获取项目背景**。

**Q：CLAUDE.md 需要经常更新吗？**

A：项目定位、团队、技术栈这些很少变。文档结构和核心项目索引可能每周微调一下。不需要每天更新，但如果项目有大的方向变化（比如技术栈迁移、团队扩编），记得同步。

**Q：团队其他人也能看到这个文件吗？**

A：CLAUDE.md 就是一个普通的 Markdown 文件，放在项目里，所有有仓库权限的人都能看。这其实是一个额外的好处——它同时也是一份给新人的项目速查手册。
