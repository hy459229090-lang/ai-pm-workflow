# 第 04 篇：让 AI 当你的数据分析师 — 数据取数自动化

> 一句话：不是"教 AI 写 SQL"，是让它成为你的数据知识库 + 分析师 + 可视化专家。

**上一篇**：[[03-skills|第 03 篇：把重复工作变成一句话 — Skill 与原子化能力]]

---

## 一个反直觉的发现

过去这一年多，我一直在尝试用 AI 解决数据问题。毛球 App 上线 MVP 后，DAU 到了 12 万，老板开始频繁问各种数据：

"昨天的 DAU 怎么样？""社区发布率涨了吗？""附近服务的转化率多少？"

一开始我每次都是手动写 SQL 去查——`t_event_log` 里捞登录数据，`t_booking` 里看预约转化。后来发现同样的问题会被问第二遍、第三遍，每次都要重新想表结构和字段。

试过各种方案——从最早的"帮我看下这个表有什么数据"，到后来让 AI 直接生成 SQL、跑分析、出报表。

大部分尝试都失败了——要么 AI 理解不了业务口径，要么生成的东西没法直接用。

但有一个 Skill，我坚持用了一年多，而且用得越来越多：**数据取数自动化。**

一开始我只是想"省点事"——不用每次都跟 AI 描述毛球的表结构（`t_pet` 是宠物档案、`t_event_log` 是行为日志、`user_id < 1000` 是测试数据）。

但用了一年多后，我发现它带来的价值远超预期：

- **验证数据**：毛球首页从时间线改成推荐 feed 后，上线前看转化漏斗和功能渗透率
- **收集数据**：周报月报要的核心指标（DAU、留存、发布率），老板突然问的"宠物档案完善率多少"
- **产品分析**：用户行为路径（浏览帖子 → 点赞 → 关注）、功能使用情况、异常波动定位
- **快速汇报**：临时被叫去开会，5 分钟拉出毛球社区和商城的关键数据

它不再是一个"取数工具"，而是一个**全职数据分析师**。

这个过程我实践了一年多，效果很好——AI 对毛球表结构的理解比很多新入职的同事还快。

---

## 它是怎么工作的

我的 `/data` Skill 不是简单的"SQL 生成器"。

它是一套完整的数据知识体系 + 分析能力：

```
用户问："毛球过去 7 天的 DAU 怎么样？"

↓

AI 内部执行：
1. 识别指标：DAU = 日活跃用户数
2. 查数据字典：用户行为日志在 t_event_log，user_id 标识用户
3. 应用业务规则：排除测试账号 (user_id < 1000)、只算正式环境
4. 生成 SQL → 执行 → 格式化输出
5. 可选：生成趋势图、对比上周、计算环比
```

**关键不是 AI 会写 SQL——是它理解毛球的业务数据。**

这是整个系统的地基。

### 我的数据字典长这样（毛球 App）

### 数据字典怎么来？

**不是手动写的——是 AI 自己分析的。**

我的做法：

1. 让 AI 连接数据库，`DESCRIBE` 每张表
2. 采样前 100 行数据，让 AI 推断字段含义
3. AI 生成初步字典，我 review 一遍
4. 业务口径补充（比如"user_id < 1000 是测试数据"这种规则）

## SQL 规则：业务口径的沉淀

光有数据字典不够——**AI 写的 SQL 要符合业务口径。**

### 我的规则示例

```markdown
## 默认过滤规则
- 排除测试数据：user_id >= 1000
- 只算正式环境：env = 'prod'
- 时间范围：默认近 7 天，除非用户指定

## 指标定义（毛球业务口径）
- DAU：当天有登录行为的去重用户数（用 last_login_time 判断）
- 新增用户：created_at 在当天范围内的用户
- 内容发布率：DAU 中当天发布过内容的用户占比
- 服务预约转化率：浏览过附近服务页面的用户中完成预约的占比
- 留存：次日/7 日/30 日，用登录行为判断而非注册

## 输出格式
- Markdown 表格
- 按时间倒序
- 超过 100 行时只展示汇总
```

### 规则怎么来？

**跟 AI 对话对话出来的。**

最开始我没有规则，AI 生成的 SQL 经常不符合预期：

- 包含了测试数据 → 加规则 `user_id >= 1000`
- 时间范围不对 → 加规则"默认近 7 天"
- 输出太乱 → 加格式规范

每次发现问题，我就跟 AI 说"这样不对，应该是 XXX"，然后让它更新规则。

迭代了大概十几轮，现在稳定下来的规则覆盖了 90% 的场景。

## 当 AI 口径和业务口径不一致时

这是最大的坑。

### 问题

毛球的业务世界里有很多"中间表"——不是原始日志，是经过加工聚合的。

比如：
- 一张"每日活跃用户表"，已经聚合好了 DAU/MAU
- 一张"用户画像表"，打标了宠物品种偏好、消费等级等属性

这些表的口径是业务定义的，AI 直接看原始数据推导不出来。

### 解法

**先解释逻辑，再执行。**

我现在的工作流：

1. 用户问毛球的复杂指标（比如"养猫用户的内容发布率"）
2. AI 先输出它对指标的理解："我理解养猫用户是 t_pet.species = 'cat' 的用户，发布率是当天有发帖行为的 DAU 占比，对吗？"
3. 我确认/纠正
4. AI 生成 SQL 执行

**多这一步，能避免 99% 的逻辑错误。**

而且这个对话过程本身也是知识沉淀——AI 记住这次校准，下次就懂了。

## 搞不定的场景

虽然我用了这么久，但 `/data` 不是万能的。

### 1. 中间表没有权限

AI 只能访问它有权访问的表。毛球的核心推荐数据在数仓的 `dw_recommend_log` 表里，AI 只能连业务库的 `t_event_log`。那就跨不过去。

**变通**：我手动导出中间表，丢给 AI 分析。

### 2. 自己都不了解表的时候

有时候业务表是别人建的、文档缺失、连字段名都是拼音缩写。

这时候 AI 也懵——它只能基于已有信息推断。

**变通**：先让 AI 采样数据、推断含义，再找建表的人确认。这个过程本身也是在帮你理解数据。

### 3. 极度复杂的连表

超过 5 张表 JOIN、嵌套多层子查询、各种 CASE WHEN——这种 SQL AI 能写，但执行效率可能不如手写优化的。

**变通**：让 AI 写基础版本，我手动优化。

## 人人都是数据科学家？

这是我写这篇最想讨论的。

### 我的观察

"人人都是数据科学家"这个口号喊了很多年，但一直没实现。

原因是**工具门槛一直存在**——SQL 要学、数据模型要懂、可视化工具要熟练。

但 AI 把这个门槛抹平了。

现在的情况是：
- 你不用会写 SQL——用自然语言描述，AI 生成
- 你不用懂数据模型——数据字典 AI 维护
- 你不用学可视化——AI 直接输出图表

**唯一需要的能力是：知道要问什么问题。**

### 但这里有争议

有人会说："AI 生成的 SQL 你敢直接用吗？""口径不一致怎么办？""数据准确性谁负责？"

我的回答：

1. **不敢直接用→让 AI 先解释逻辑**（上面讲了）
2. **口径不一致→对话校准，AI 记住**（这也是知识沉淀）
3. **准确性→最终责任还是人**（AI 是工具，决策是你）

**"人人都是数据科学家"不是说不需要专家了——是说工具足够好用，让非专家也能做 80 分的数据分析。**

剩下的 20 分（复杂建模、数据治理、口径统一）还是需要专家，但日常分析需求 80% 都能自己搞定。

### 裂口

我很好奇你在用什么方式做数据分析？

- 还是手动写 SQL 吗？
- 有试过让 AI 生成吗？效果怎么样？
- 你觉得"人人都是数据科学家"是机会还是坑？

## 自查清单

- [ ] 数据字典是 AI 分析 + 人工 review 的，不是手动写的
- [ ] SQL 规则是从真实对话中沉淀的，不是一开始设计好的
- [ ] 复杂指标先解释逻辑再生成，避免口径偏差
- [ ] 承认局限性——没有权限的表、极度复杂的 JOIN 搞不定
- [ ] 最终责任在人——AI 是工具，决策和验证还是你
