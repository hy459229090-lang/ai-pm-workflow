# 为什么你的 AI 只会说废话？因为它没有"技能"

## 一个让我崩溃的重复

搭完项目配置（CLAUDE.md）和文档地图（WORKSPACE_MAP）之后，我的 AI 确实好用多了——理解项目背景、能自己找文件、产出不再是通用废话。

但用了两周，我发现自己在反复说同一句话。

"帮我查一下昨天的 DAU，数据在 t_event_log 表，用 event_time 过滤，排除 user_id 小于 1000 的测试账号，输出按日期倒序。"

周一说一遍。周三说一遍。周五又说一遍。每次 30 秒描述一大段。

AI 明明知道我的项目，知道文件在哪——但**它不会做这件事**。每次都要我手把手教。

这就像你有个同事，人挺聪明，了解业务——但你让他做任何具体的事，他都要你从头教一遍。你会疯的。

---

## 我犯的最大错误：把 Skill 当成 Prompt 模板

我最初的做法是写了一堆"Prompt 模板"——当用户说 X 的时候，AI 按 Y 的格式回答。

这是错的。

模板的问题是**AI 变成了填空机器**。我定义了取数的每一步——用什么表、怎么过滤、输出什么格式——AI 严格照做，产出千篇一律。简单查询还行，稍微复杂一点的需求，它就不知道变通了。

后来我想明白了一件事：

**Skill 不是操作手册，是能力。**

操作手册是"照着念"——人不需要理解，按步骤执行。
能力是"学会了"——理解原理和约束，遇到新情况自己判断。

这跟你招新人是一回事。你是给他一份 SOP 让他照着做，还是教会他判断标准让他自己干？前者能应对已知场景，后者能应对所有场景。

---

## 能力定义 vs. 模板定义

拿取数举例：

**模板式**（早期的错误写法）：
```
当用户说"查 DAU"时：
1. 使用 t_event_log 表
2. 按 event_time 过滤
3. COUNT(DISTINCT user_id)
4. 排除 user_id < 1000
5. 输出 Markdown 表格
```

**能力式**（现在的写法）：
```
你现在具备数据查询能力。
数据源：PostgreSQL，表结构见 docs/data/数据字典.md
默认规则：近 7 天、排除测试数据、Markdown 表格输出
常见查询模式：DAU、留存、转化率...
```

区别在哪？模板式告诉 AI"这个场景怎么做"，能力式告诉 AI"你会做这类事了，具体怎么做你自己判断"。

结果是：用模板，我只能查预定义的几种数据；用能力定义，AI 能处理我没想到的查询需求，因为它理解了数据源和规则，可以**自己组合**。

## 我实际在用的三个核心 Skill

### `/data` — 说人话查数据

以前："帮我查过去 7 天 DAU，表是 t_event_log，字段是 user_id 和 event_time，过滤条件是..."（30 秒）

现在："/data 过去 7 天 DAU"（3 秒）

AI 知道用什么表、怎么过滤、怎么格式化。遇到没见过的查询，它会查数据字典自己拼 SQL。

### `/draw` — 一句话出概念图

这个 Skill 让 AI 有了"画图"的能力。它知道调哪个 API、什么模型适合什么场景、保存到哪里。

关键细节：我在 Skill 里加了一条"生成前必须先优化 prompt 让我确认"——这是踩坑后加的。用户随口描述和适合图片生成的 prompt 差距很大，不优化直接生成十有八九要重来。

### PRD 写作 — 按标准写，不是按模板填

最早的 PRD Skill 定义了每一段写什么，AI 产出的 PRD 千篇一律。后来我改成只定义**质量标准**——什么字段必须有、什么要自检、格式约束是什么——让 AI 根据输入自己组织内容。

效果差了一个量级。同样的需求输入，能力式的产出能直接进评审，模板式的还要改半小时。

## 什么时候 Skill 不够用

这可能是这篇最有价值的部分。

当我理顺了工作流之后，我发现了一个新问题：**有些任务不是一个能力能搞定的。**

一个需求要出 PRD、要出技术方案、要出埋点方案。这三份文档有依赖关系——埋点方案要基于 PRD 里的功能定义，技术方案要回应 PRD 里的约束。

更关键的是，**我没法同时关注所有环节的细节**。PRD 写得好不好、技术方案可行不可行、埋点有没有遗漏——我的精力和注意力覆盖不了。

这是我开始琢磨 Agent Team 的原点。一个人做不完的事，那就找一群人来做——只不过这群"人"是 AI Agent。

当时想明白这件事的时候，真的很兴奋。凌晨一点还在跟朋友分享："我搞出来一个东西，14 个 AI 帮我干活，我只管拍板。"

但 Agent Team 是后面的话题（L5 会详细讲）。现在只需要记住一个判断标准：

- **流程固定、一次出结果** → 用 Skill
- **多份产出、需要交叉审视、你 review 不过来** → 拉 Team
- **一次性的创造性工作** → 直接跟 AI 对话就行

这个边界说实话我也还在摸索。有时候以为 Skill 够用，做到一半发现得拉 Team。也有觉得需要 Team 的，后来发现一个好的 Skill 就搞定了。

## 你的第一个 Skill 怎么写

1. **找到你的重复操作**：回忆过去一周，你跟 AI 重复说了什么？
2. **提取共性**：这些操作的固定部分是什么（数据源、工具、格式）？
3. **写能力定义**：在 `.claude/commands/` 或 `.cursor/skills/` 创建文件。只定义能力和约束，不定义每一步怎么做。
4. **迭代 2-3 轮**：用真实任务测试，补充缺失的规则，删掉多余的限制。
