# 14 个 AI Agent 协作，我是怎么设计的

> 本文是我用 Agent Team 工作流实战 3 个月的实践沉淀。不是教程，是一个真实工作流从混乱到秩序的演化过程。

---

## 一、为什么需要 Agent Team

第 03 篇讲了 Skill——把重复操作封装成原子化能力。
第 04 篇讲了数据取数——让 AI 成为你的全职数据分析师。

但用了半年 Skill，我发现了一个新问题：

**有些任务，一个 AI 真的搞不定。**

不是它不够聪明，是任务本身就涉及多份产出、多个视角、多轮评审。

举个例子：

毛球 App 要做首页推荐 feed 改版（从时间线改成个性化推荐），需要：
- 一份 PRD（产品需求文档）—— 推荐策略、排序逻辑、冷启动方案
- 一份技术方案（后端 + 算法）—— 召回模型、特征工程、AB 测试框架
- 一份埋点方案（数据追踪）—— 曝光/点击/停留/负反馈
- 一份测试用例（QA 视角）—— 推荐结果多样性、极端场景兜底

这四份文档有依赖关系——技术方案要回应 PRD 里的约束，埋点方案要基于功能定义，测试用例要覆盖所有场景。

更关键的是，**我没法同时关注所有环节的细节**。

PRD 里推荐策略写得好不好、技术方案里的召回模型可行不可行、埋点有没有遗漏负反馈事件——我的精力和注意力覆盖不了。

这时候，一个 Skill 不够用了。我需要拉一个**Agent Team**。

---

## 二、Agent Team 是什么：不是多个 AI，是一套协作机制

最初我理解的 Agent Team 是"多叫几个 AI 一起干活"。

这个理解是错的，或者说太浅了。

**Agent Team 真正的意义是一套协作机制。** 就像你带团队，不是把一群人拉进一个群，然后说"你们自己干"。

你需要：
- 明确分工（谁负责什么）
- 定义输入输出（交付标准是什么）
- 设置检查点（什么时候 review）
- 处理冲突（意见不一致听谁的）

Agent Team 也是一样的逻辑。

## 三、我的第一个 Agent Team：14 个 AI 的混乱与秩序

说一个真实的故事。

去年 11 月，毛球 App 要做首页推荐 feed 改版——从时间线改成个性化推荐。

任务拆解出来有：
- 产品分析（竞品分析：小红书/抖音的推荐逻辑、毛球用户画像、社区内容分类）
- 需求定义（PRD：推荐策略、排序逻辑、冷启动方案、负反馈机制）
- 技术方案（召回模型：协同过滤 + 内容召回、特征工程、AB 测试框架）
- 数据埋点（曝光/点击/停留时长/不感兴趣按钮、推荐效果追踪）
- 测试用例（推荐多样性、新用户冷启动、极端场景兜底）

我一个人肯定干不完——就算每个环节都用 Skill，也得花几周。

于是我拉了一个 Agent Team，最多时有 14 个 AI 同时在线。

### 3.1 第一版设计：完全分布式

我把 14 个 AI 拉进一个群（其实是多个对话线程），然后说：

"产品分析组先输出毛球竞品分析，然后需求组基于分析写推荐 PRD，技术组基于 PRD 设计推荐方案..."

结果是一片混乱：
- 产品分析组输出了小红书/抖音的竞品对比，需求组没注意到，还在等我手动转发
- 技术组写的召回方案和 PRD 里的冷启动约束对不上
- 数据组和测试组发现推荐负反馈的定义有歧义，但不知道问谁
- 我每隔 5 分钟要切换一次对话，确认进度、解决冲突

**那一天真的累。14 个 AI 给我干活，我比我自己干还累。**

晚上 11 点，我盯着屏幕上乱七八糟的对话，意识到一个问题：

我不是在管团队，我是在当**人肉路由器**——每个信息都要经过我，每个冲突都要我来裁决。

这不可持续。

## 四、第二版设计：主从架构 + 评审机制

第二天我重新设计了架构。

### 4.1 核心改变

**1. 设一个队长（Lead Agent）**
- 负责统筹全局、分发任务、收集产出
- 我只跟队长对话，不跟组员直接对话

**2. 定义清晰的输入输出**
- 每个 Agent 只负责一个环节
- 输入是什么、输出是什么、质量标准是什么

**3. 设置评审点**
- 关键产出完成后，自动触发评审
- 评审不通过，打回重做

**4. 冲突解决机制**
- 技术组和产品组意见不一致？
- 先各自陈述理由，然后由评审组裁决

### 4.2 效果立竿见影

那天之后，我只需要做三件事：
1. 跟队长说清楚要做什么
2. 等队长通知"初稿完成"
3. Review 关键节点，给出反馈

剩下的交给 Team。

## 五、什么时候需要拉 Agent Team

这可能是这篇最有价值的部分。

### 5.1 我的判断标准

| 任务特征 | 用 Skill | 拉 Team |
|----------|----------|--------|
| 产出数量 | 1 份 | 多份（≥3） |
| 依赖关系 | 无/简单 | 复杂（A 依赖 B） |
| 视角需求 | 单一 | 多个（产品/技术/用户/商业） |
| 你的精力 | 能覆盖全部细节 | 覆盖不了，需要分权 |

### 5.2 简单说

- **流程固定、一次出结果** → 用 Skill
- **多份产出、需要交叉审视、你 review 不过来** → 拉 Team
- **一次性的创造性工作** → 直接跟 AI 对话就行

这个边界说实话我也还在摸索。

有时候以为 Skill 够用，做到一半发现得拉 Team。也有觉得需要 Team 的，后来发现一个好的 Skill 就搞定了。

## 六、怎么写一个 Agent Team

以毛球 App 的"需求分析"Team 为例。

### 6.1 第 1 步：定义任务边界

这个 Team 要做什么：
- 输入：一个产品想法/需求
- 输出：PRD + 技术方案 + 埋点方案 + 测试用例

### 6.2 第 2 步：设计角色分工

| 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 产品 Agent | 写 PRD | 产品想法 | PRD 文档 |
| 技术 Agent | 设计技术方案 | PRD | 技术方案 |
| 数据 Agent | 设计埋点方案 | PRD + 技术方案 | 埋点文档 |
| 测试 Agent | 写测试用例 | PRD + 技术方案 | 测试用例 |
| 评审 Agent | 审查所有产出 | 所有文档 | 评审意见 |
| 队长 Agent | 统筹分发 | 你的需求 | 最终交付物 |

### 6.3 第 3 步：定义协作流程

```
你 → 队长 Agent：需求描述
队长 → 产品 Agent：写 PRD
产品 Agent → 评审 Agent：PRD 评审
评审 Agent → 队长：评审通过/打回
队长 → 技术 Agent + 数据 Agent + 测试 Agent：并行工作
各 Agent → 队长：交付
队长 → 你：最终交付物
```

### 6.4 第 4 步：定义质量标准

每个环节产出前，定义清楚：
- 必填字段有哪些
- 格式规范是什么
- 自检清单是什么

### 6.5 第 5 步：测试和迭代

用几次真实任务测试。你会发现：
- 哪些角色多余（可以合并）
- 哪些环节缺失（需要补充）
- 哪些标准不够细（需要完善）

我的经验是迭代 3-5 轮才稳定。

## 七、踩过的坑

### 7.1 坑 1：队长太忙，变成瓶颈

第一版设计里，队长要统筹全局、分发任务、收集产出、解决冲突——累死了。

**解法**：
- 队长只负责分发和收集，不负责审查
- 审查交给评审 Agent
- 冲突由评审组裁决，队长执行

### 7.2 坑 2：评审流于形式

评审 Agent 最开始只会说"看起来不错"——没有实际价值。

**解法**：
- 定义评审清单（必须检查的 10 个点）
- 评审意见必须包含"通过/打回" + 具体理由
- 打回次数多了，评审 Agent 会学习

### 7.3 坑 3：产出风格不一致

产品 Agent 写的 PRD 和技术 Agent 写的方案风格差异太大，阅读体验很差。

**解法**：
- 定义统一的写作风格指南
- 提供模板（但不规定内容）
- 队长最后统一格式

## 八、效果对比

| 指标 | 我一个人干 | 用 Skill | 拉 Agent Team |
|------|------------|----------|---------------|
| 时间 | 3-5 天 | 1-2 天 | 2-4 小时 |
| 我的精力投入 | 100% | 30% | 10% |
| 产出质量 | 看状态 | 稳定 | 稳定且全面 |
| 覆盖视角 | 产品 | 产品 + 数据 | 产品 + 技术 + 数据 + 测试 |

## 九、核心洞察

这一年多折腾下来，我最大的收获：

**Skill 是给 AI 增加能力，Agent Team 是给自己减少负担。**

Skill 让你不用重复说同一句话。
Agent Team 让你不用同时关注所有细节。

但 Agent Team 不是银弹——它解决的是"多份产出、多视角、你 review 不过来"的问题。

如果只是一个简单任务，用 Skill 就够了，拉 Team 反而是浪费。
