# AI 工具的正确用法不是写 Prompt，是给它"原子化能力"

> 适合知乎专栏，也可回答"Claude Code 怎么配置""AI 工作流怎么搭建""Cursor 的 Skills 怎么用"等问题。

---

## 结论

把 AI 的重复操作封装成 Skill（技能），但不要写成 Prompt 模板，而是写成**能力定义**。区别在于：模板让 AI 照着做，能力定义让 AI 学会做。前者应对已知场景，后者应对所有场景。

这篇讲的是单个 Skill 的设计。当任务复杂到一个 Skill 搞不定时，需要 Agent Team——但那是另一个话题。

---

## 问题：AI 有记忆但没有技能

如果你跟着这个系列走，你已经有了 CLAUDE.md（让 AI 理解项目）和 WORKSPACE_MAP（让 AI 找到文件）。AI 的产出质量已经比裸用好了一个量级。

但你很快会发现一个新问题：**同一句话你在反复说。**

"帮我查 DAU，用 t_event_log 表，按 event_time 过滤，排除测试账号，输出 Markdown 表格……"

每次 30 秒。每周说 3-5 次。一个月下来你在"教 AI 做同一件事"上花了几个小时。

根本原因：AI 理解你的项目（上下文），但不具备**执行具体任务的能力**（技能）。每次执行都要你从头描述流程。

---

## Skill 的正确理解：能力，不是模板

大多数人（包括早期的我）把 Skill 理解成 Prompt 模板：

```
当用户说"查 DAU"时，执行以下 SQL：
SELECT date, COUNT(DISTINCT user_id) FROM t_event_log WHERE ...
```

这么做有一个严重问题：**AI 变成了填空机器。** 它严格执行预定义的步骤，无法处理模板没覆盖的场景。你预定义了查 DAU 的模板，用户问"各城市的 7 日留存趋势"——超纲了，AI 不知道怎么办。

正确的做法是写**能力定义**：

```markdown
# 数据查询能力

你现在具备数据查询能力。

## 数据源
- PostgreSQL 业务库，表结构见 docs/data/数据字典.md
- ClickHouse 分析库，用于大数据量聚合

## 规则
- 默认近 7 天，可通过自然语言指定时间范围
- 排除测试数据（user_id >= 1000）
- 输出 Markdown 表格，按时间倒序

## 参考模式
- DAU、MAU、留存、转化率等常见查询模式见数据字典
```

关键区别：

| 维度 | 模板式 | 能力式 |
|------|--------|--------|
| AI 的角色 | 执行器（照做） | 技能者（理解后自主执行） |
| 覆盖范围 | 只有预定义的场景 | 所有相关场景 |
| 遇到新需求 | 报错或产出垃圾 | 基于规则自主组合 |
| 产出多样性 | 千篇一律 | 根据输入自适应 |

我在实际使用中验证过这个差距：同样的数据 Skill，模板式只能处理 5 种预定义查询，能力式能处理我没想到的任意查询——因为它理解了数据源和规则，能自己组合 SQL。

## 我的三个核心 Skill 及踩坑经验

### 1. 数据取数（`/data`）

用之前：每次查数据花 30 秒描述表结构和过滤条件。
用之后：`/data 过去7天DAU` — 3 秒搞定。

**踩坑**：最初没把数据字典的路径写进 Skill，AI 每次都用错误的表名。加了数据字典引用后，准确率从 60% 跳到 95%+。

### 2. 图片生成（`/draw`）

定义了 API 地址、可用模型、模型特点、保存路径。

**关键发现**：我后来在 Skill 里加了一条"生成前必须先优化 prompt 并让用户确认"。原因是用户的自然语言描述和图片生成 API 需要的 prompt 差距很大。不加这一步，十有八九要重新生成。

### 3. PRD 写作

**最大的教训**：早期版本定义了 PRD 的每一段写什么——背景写什么、目标写什么、方案写什么。AI 产出的所有 PRD 长得一模一样。

后来我删掉了具体段落要求，只保留：
- 必须包含的字段（背景、目标、方案、排期、风险）
- 质量标准（可自检的清单）
- 格式约束

效果差了一个量级。AI 开始根据不同需求自主组织内容结构，产出的 PRD 能直接进评审。

## Skill 与 Agent Team 的边界

一个经常被问的问题：什么时候用 Skill 就够了，什么时候需要更复杂的协作方式？

我的判断框架：

| 信号 | 用 Skill | 拉 Agent Team |
|------|---------|---------------|
| 产出物数量 | 1 份 | ≥2 份且有依赖 |
| 需要的视角 | 单一 | 需要技术/用户/商业多角度 |
| Review 负担 | 你能一次看完 | 你看不完，需要有人先审一轮 |
| 流程确定性 | 流程固定 | 需要中间判断和调整 |

这个边界我也在持续摸索。有时以为 Skill 够用，做到一半发现需要拉 Team；也有觉得要 Team 的，后来发现一个好的 Skill 就搞定了。

经验法则：**先用 Skill 试，搞不定再升级。** 不要过早引入复杂度。

## 技术实现

### Claude Code

在项目下创建 `.claude/commands/skill-name.md`，Claude Code 会自动识别为可调用的 Skill。

### Cursor

在 `.cursor/skills/skill-name/SKILL.md` 创建文件，Cursor 的 AI 会读取并具备对应能力。

### 通用原则

无论用什么工具，Skill 文件的核心结构是：
1. 这个能力是什么（一句话）
2. 涉及什么外部资源（API、数据源、文件路径）
3. 规则和约束（默认值、质量标准、边界条件）
4. 参考模式（常见场景的处理方式，但不是硬性模板）
